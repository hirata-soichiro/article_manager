## 1. ドメイン用語

### 1.1 記事（Article）

**定義**: ユーザーが保存・管理する技術記事の情報

**属性**:
- ID（id）: 記事の一意識別子
- タイトル（title）: 記事のタイトル
- URL（url）: 記事のURL
- 要約（summary）: 記事の要約テキスト
- タグ（tags）: 記事に付与されたタグの配列
- メモ（memo）: ユーザーが追加する個人的なメモ
- 作成日時（created_at）: 記事の登録日時
- 更新日時（updated_at）: 記事の最終更新日時

**ユースケース**:
- 記事の作成、閲覧、編集、削除
- 記事の検索、フィルタリング
- AI生成による記事の自動作成

**制約**:
- タイトルは必須、最大255文字
- URLは必須、有効なURL形式、最大2048文字
- 要約は必須、最大1000文字
- タグは最大10個まで
- メモはオプション、最大65535文字

### 1.2 タグ（Tag）

**定義**: 記事を分類・整理するためのラベル

**属性**:
- ID（id）: タグの一意識別子
- タグ名（name）: タグの名前（例: "Go", "Backend"）
- 記事数（article_count）: そのタグを持つ記事の数
- 作成日時（created_at）: タグの作成日時
- 更新日時（updated_at）: タグの最終更新日時

**ユースケース**:
- タグの作成、編集、削除
- タグによる記事のフィルタリング
- タグ一覧の表示

**制約**:
- タグ名は必須、最大50文字、ユニーク（重複不可）
- 使用中のタグ（article_count > 0）は削除不可

**Phase 1の実装方式**:
- 記事の`tags`フィールドはJSON配列で管理（例: `["Go", "Backend"]`）
- `tags`テーブルは参照用マスタとして存在（タグ一覧、編集、削除管理用）
- `article_tags`中間テーブルは作成しない

**タグ管理の動作**:
1. 記事作成時: `tags` JSONフィールドに配列として保存
2. タグ一覧取得時: `articles.tags`フィールドからJSONを抽出して集計
3. タグ名変更時: 該当タグを持つ全記事の`tags` JSONフィールドを更新
4. タグ削除時: 使用中（article_count > 0）の場合はエラー

**Phase 2への移行計画**:
- パフォーマンス改善のため、`article_tags`中間テーブルを導入予定
- マイグレーションで既存の`tags` JSONから中間テーブルへデータ移行
- `tags` JSONフィールドは互換性のため残す（読み取り専用）

### 1.3 書籍推薦（Book Recommendation）

**定義**: AIが生成する、記事のタグに基づいた技術書籍の推薦情報

**属性**:
- タイトル（title）: 書籍のタイトル
- 著者（authors）: 著者名の配列
- 出版社（publisher）: 出版社名
- 出版日（published_date）: 出版日
- 説明（description）: 書籍の説明
- サムネイル（thumbnail）: 書籍の表紙画像URL
- 詳細リンク（info_link）: Google Booksへのリンク

**ユースケース**:
- 記事一覧画面での書籍推薦表示
- 書籍推薦のキャッシュ管理

**制約**:
- キャッシュ有効期限は24時間
- 最大表示件数は5〜10件程度

### 1.4 書籍推薦キャッシュ（Book Recommendation Cache）

**定義**: 生成された書籍推薦を一時保存するキャッシュデータ

**属性**:
- ID（id）: キャッシュの一意識別子
- 書籍情報（books）: 書籍推薦の配列（JSON形式）
- 生成日時（generated_at）: 推薦の生成日時
- 有効期限（expires_at）: キャッシュの有効期限

**キャッシュキー生成方法**:
1. 全記事のタグを抽出
2. 重複を除いてユニークなタグリストを作成
3. タグリストをアルファベット順にソート
4. JSON文字列化
5. SHA256ハッシュ化

**キャッシュ無効化条件**:
- 記事の追加/削除でタグセットが変化した場合
- 有効期限（24時間）を過ぎた場合

## 2. ビジネス用語

### 2.1 AI生成（AI Generation）

**定義**: Gemini APIを使用して、記事のURLから自動的にタイトル、要約、タグを生成する機能

**処理フロー**:
1. ユーザーがURLを入力
2. バックエンドがURLのコンテンツを取得
3. Gemini APIで要約とタグを生成
4. 生成結果をフロントエンドに返却
5. ユーザーが確認・編集後、保存

**利点**:
- 記事登録の効率化
- 一貫性のある要約とタグ付け

### 2.2 CRUD操作

**定義**: Create（作成）、Read（読み取り）、Update（更新）、Delete（削除）の基本操作

**適用対象**:
- 記事のCRUD
- タグのCRUD

### 2.3 検索（Search）

**定義**: キーワードやタグによる記事の検索機能

**検索対象**:
- 記事のタイトル
- 記事の要約

**検索方法**:
- FULLTEXT検索（MySQL）
- タグによるフィルタリング

**特徴**:
- リアルタイム検索（300msデバウンス）
- 複数タグでのAND検索

### 2.4 フィルタリング（Filtering）

**定義**: タグによる記事の絞り込み機能

**フィルタリング方法**:
- 単一タグ選択: そのタグを持つ記事のみ表示
- 複数タグ選択: すべてのタグを持つ記事のみ表示（AND条件）

### 2.5 ページネーション（Pagination）

**定義**: 記事一覧を複数ページに分割して表示する機能

**パラメータ**:
- limit: 1ページあたりの表示件数（デフォルト20件）
- offset: スキップする件数

**Phase 1**: 実装予定（必要に応じて）

### 2.6 Phase（フェーズ）

**定義**: プロジェクトの開発段階を示す用語

**Phase 1（MVP期）**:
- **目的**: 最小限の機能で動作するプロダクトを開発
- **特徴**:
  - 認証機能なし（開発環境のみ）
  - 基本的なCRUD操作のみ実装
  - 単一ユーザー想定
  - 開発環境での運用
- **期間**: 初期6ヶ月

**Phase 2（本格運用期）**:
- **目的**: 本格的な運用に向けた機能拡張
- **特徴**:
  - JWT認証・認可の実装
  - 高度な機能追加（チーム共有、エクスポート、マルチユーザー対応など）
  - 本番環境デプロイ
  - セキュリティ強化（CORS制限、CSRF対策など）
- **期間**: 6ヶ月以降

**開発における使い方**:
- ドキュメントやコード内で「Phase 1で実装」「Phase 2で追加予定」のように記載
- 機能の実装優先度を明確にする

### 2.7 永続的ドキュメント（Persistent Document）

**定義**: アプリケーション全体の「何を作るか」「どう作るか」を定義する恒久的なドキュメント

**配置場所**: `docs/`

**特徴**:
- アプリケーションの基本設計を記述
- 頻繁に更新されない
- 大きな設計変更時のみ更新
- プロジェクト全体の「北極星」として機能

**種類**:
- `product-requirements.md` - プロダクト要求定義書
- `functional-design.md` - 機能設計書
- `architecture.md` - 技術仕様書
- `repository-structure.md` - リポジトリ構造定義書
- `development-guidelines.md` - 開発ガイドライン
- `glossary.md` - ユビキタス言語定義

### 2.8 ステアリングファイル（Steering File）

**定義**: 特定の開発作業における「今回何をするか」を定義する一時的なドキュメント

**配置場所**: `.steering/[YYYYMMDD]-[開発タイトル]/`

**命名規則**: `.steering/YYYYMMDD-開発タイトル/`
- 例: `.steering/20250103-initial-implementation/`
- 例: `.steering/20250115-add-tag-feature/`

**構成**:
- `requirements.md` - 今回の作業の要求内容
  - 変更・追加する機能の説明
  - ユーザーストーリー
  - 受け入れ条件
  - 制約事項
- `design.md` - 変更内容の設計
  - 実装アプローチ
  - 変更するコンポーネント
  - データ構造の変更
  - 影響範囲の分析
- `tasklist.md` - タスクリスト
  - 具体的な実装タスク
  - タスクの進捗状況
  - 完了条件

**特徴**:
- 特定の作業・変更に特化
- 作業ごとに新しいディレクトリを作成
- 作業完了後は履歴として保持
- 変更の意図と経緯を記録

**永続的ドキュメントとの違い**:
- 永続的ドキュメント: アプリケーション全体の設計（What/How to build）
- ステアリングファイル: 今回の作業の内容（What to do now）

## 3. UI/UX用語

### 3.1 記事カード（Article Card）

**定義**: 記事一覧画面で使用される、記事情報を表示するカードコンポーネント

**表示内容**:
- 記事タイトル
- 記事要約（短縮版）
- タグ一覧
- 作成日時
- 編集・削除ボタン

**インタラクション**:
- カードクリック: 記事詳細画面へ遷移
- 編集ボタン: 記事編集画面へ遷移
- 削除ボタン: 確認ダイアログ表示後、削除実行

### 3.2 検索バー（Search Bar）

**定義**: キーワード入力による記事検索を行うUIコンポーネント

**機能**:
- キーワード入力
- デバウンス処理（300ms）
- 検索実行

### 3.3 タグフィルター（Tag Filter）

**定義**: タグボタンによる記事のフィルタリングを行うUIコンポーネント

**機能**:
- タグ一覧表示
- タグの選択/解除
- 選択中のタグをハイライト表示
- 記事数表示

### 3.4 トースト通知（Toast Notification）

**定義**: 操作結果をユーザーに通知する一時的なメッセージ表示

**種類**:
- 成功（success）: 緑色、操作成功を通知
- エラー（error）: 赤色、エラー発生を通知
- 警告（warning）: 黄色、警告メッセージを表示
- 情報（info）: 青色、情報を表示

**表示位置**: 画面右上

**表示時間**: デフォルト3秒（自動で非表示）

### 3.5 ローディングスピナー（Loading Spinner）

**定義**: 処理中であることをユーザーに示すアニメーション表示

**表示タイミング**:
- AI生成処理中
- データ取得中
- 保存処理中

### 3.6 フォームバリデーション（Form Validation）

**定義**: 入力値の妥当性をチェックする機能

**バリデーション方式**:
- クライアント側: リアルタイムバリデーション
- サーバー側: 最終バリデーション

**エラー表示**: フィールド直下に赤色でエラーメッセージを表示

## 4. 技術用語

### 4.1 Clean Architecture

**定義**: 依存関係を内側に向けることで、ビジネスロジックを外部の詳細から独立させる設計パターン

**レイヤー構成**:
1. **Domain（ドメイン層）**: エンタープライズビジネスルール
2. **UseCase（ユースケース層）**: アプリケーションビジネスルール
3. **Interface（インターフェース層）**: HTTPハンドラー
4. **Infrastructure（インフラ層）**: 外部システム統合

**依存関係の方向**: Infrastructure → Interface → UseCase → Domain

### 4.2 Repository Pattern

**定義**: データアクセスを抽象化し、ビジネスロジックからデータソースの詳細を隠蔽するパターン

**実装**:
- Interfaceで定義: `domain/repository/`
- 実装: `infrastructure/repository/`

**利点**:
- テスタビリティの向上（モック化が容易）
- データソースの切り替えが容易

### 4.3 Dependency Injection（依存性注入）

**定義**: オブジェクトの依存関係を外部から注入する設計パターン

**実装場所**: `cmd/server/main.go`

**例**:
```go
repo := repository.NewArticleRepository(db)
useCase := usecase.NewArticleUseCase(repo)
handler := handler.NewArticleHandler(useCase)
```

### 4.4 ミドルウェア（Middleware）

**定義**: HTTPリクエスト/レスポンスの処理を横断的に行うコンポーネント

**Phase 1のミドルウェア**:
- CORS（クロスオリジン制御）
- Recovery（パニックリカバリー）
- Logger（リクエスト/レスポンスログ）

**Phase 2のミドルウェア**:
- Authentication（認証）
- Authorization（認可）
- RateLimit（レート制限）

### 4.5 サーキットブレーカー（Circuit Breaker）

**定義**: 外部API障害時の連鎖障害を防ぐパターン

**状態**:
- **Closed（正常）**: リクエストを通常通り送信
- **Open（開放）**: リクエストを遮断、フォールバック処理を実行
- **Half-Open（半開）**: 試験的にリクエストを送信、成功すればClosedへ遷移

**適用対象**:
- Gemini API
- Google Books API

### 4.6 レート制限（Rate Limiting）

**定義**: 一定期間内のリクエスト数を制限する機能

**実装方式**: Token Bucket方式

**適用対象**:
- Gemini API: 10リクエスト/秒
- Google Books API: 1リクエスト/秒

### 4.7 リトライ（Retry）

**定義**: 失敗したリクエストを再試行する機能

**リトライ戦略**:
- 最大リトライ回数: 3回
- 指数バックオフ: 1秒 → 2秒 → 4秒
- ジッター: ランダムな遅延を追加

**リトライ対象エラー**:
- ネットワークタイムアウト
- 5xx系サーバーエラー
- レート制限エラー（429）

### 4.8 フォールバック（Fallback）

**定義**: エラー発生時の代替処理

**適用例**:
- Gemini API失敗時: 空の要約・タグを返却、手動入力を促す
- Google Books API失敗時: キャッシュから返却、または空配列を返却

### 4.9 FULLTEXT検索

**定義**: MySQLのFULLTEXTインデックスを使用した全文検索

**対象フィールド**:
- articles.title
- articles.summary

**インデックス定義**:
```sql
FULLTEXT INDEX ft_idx_title_summary (title, summary)
```

### 4.10 トランザクション（Transaction）

**定義**: 複数のデータベース操作を一つの処理単位として扱う仕組み

**ACID特性**:
- Atomicity（原子性）: すべて成功するか、すべて失敗するか
- Consistency（一貫性）: データの整合性を保つ
- Isolation（隔離性）: 他のトランザクションから独立
- Durability（永続性）: コミット後はデータが永続化

**適用ケース**:
- 記事削除と関連データの削除
- タグ名変更と記事のタグ更新

### 4.11 マイグレーション（Migration）

**定義**: データベーススキーマの変更をバージョン管理する仕組み

**実装方式**: 組み込み型マイグレーションシステム

**マイグレーションファイル**: `api/internal/infrastructure/database/migrations/`

**実行タイミング**: アプリケーション起動時に自動実行

### 4.12 ホットリロード（Hot Reload）

**定義**: コード変更時に自動でアプリケーションを再起動する機能

**ツール**:
- バックエンド: Air
- フロントエンド: Next.js開発サーバー（npm run dev）

### 4.13 JWT（JSON Web Token）

**定義**: JSON形式のトークンベース認証方式（Phase 2で実装予定）

**構造**:
- **Header（ヘッダー）**: アルゴリズムとトークンタイプ（例: `{"alg": "HS256", "typ": "JWT"}`）
- **Payload（ペイロード）**: ユーザー情報とクレーム（例: `{"user_id": 123, "email": "user@example.com", "role": "user", "exp": 1642345678}`）
- **Signature（署名）**: ヘッダーとペイロードの署名（改ざん防止）

**トークンの種類**:
- **Access Token**: 有効期限24時間、API認証に使用
- **Refresh Token**: 有効期限30日、Access Token再発行に使用（Phase 2.1で実装）

**保存場所**:
- Phase 2: localStorage（XSS対策としてhttpOnly Cookieも検討）

**利点**:
- ステートレス認証（サーバー側でセッション管理不要）
- スケーラブル（複数サーバーで共有可能）
- モバイルアプリとの連携が容易

### 4.14 CORS（Cross-Origin Resource Sharing）

**定義**: 異なるオリジン（ドメイン・ポート・プロトコル）間でのリソース共有を制御するセキュリティ機構

**オリジンの例**:
- `http://localhost:3000` と `http://localhost:8080` は異なるオリジン（ポートが異なる）
- `https://example.com` と `http://example.com` は異なるオリジン（プロトコルが異なる）

**Phase 1の設定**:
- すべてのオリジンを許可（`Access-Control-Allow-Origin: *`）
- 開発環境のため、制限なし

**Phase 2の設定**:
- 特定ドメインのみ許可（例: `https://example.com`, `https://www.example.com`）
- 許可するメソッド: `GET`, `POST`, `PUT`, `DELETE`
- 許可するヘッダー: `Authorization`, `Content-Type`
- 認証情報の送信: `credentials: true`

**ミドルウェア実装**: バックエンドAPIでCORSミドルウェアを設定

### 4.15 CSRF（Cross-Site Request Forgery）

**定義**: ユーザーの意図しないリクエストを攻撃者が送信させる攻撃手法

**攻撃シナリオ例**:
1. ユーザーがアプリケーションにログイン済み
2. 攻撃者が罠サイトを用意
3. ユーザーが罠サイトを訪問
4. 罠サイトがユーザーのブラウザからアプリケーションにリクエストを送信
5. ブラウザは自動的にCookieを送信（認証済みとみなされる）
6. アプリケーションは正規のリクエストと判断して処理

**Phase 1の状態**:
- 認証機能がないため、CSRF対策は未実装
- リスク: 低（認証がないため）

**Phase 2の対策**:
- **CSRFトークン**: リクエストごとに一意なトークンを生成・検証
- **SameSite Cookie属性**: `SameSite=Strict`または`SameSite=Lax`を設定
- **ダブルサブミットクッキーパターン**: CookieとHTTPヘッダーの両方でトークンを送信

**実装方法**:
- CSRFトークンをCookieに保存
- フロントエンドがリクエスト時に`X-CSRF-Token`ヘッダーにトークンを設定
- バックエンドがCookieとヘッダーのトークンを照合

### 4.16 XSS（Cross-Site Scripting）

**定義**: 悪意のあるスクリプトをWebページに注入し、他のユーザーのブラウザで実行させる攻撃手法

**攻撃タイプ**:
- **Stored XSS（格納型）**: データベースに悪意のあるスクリプトを保存し、他のユーザーが閲覧時に実行
- **Reflected XSS（反射型）**: URLパラメータなどに含まれるスクリプトを即座に実行
- **DOM-based XSS**: JavaScriptによるDOM操作の脆弱性を利用

**Phase 1の対策**:
- **Reactの自動エスケープ**: Reactは`{変数}`の形式で自動的にHTMLエスケープ
- **`dangerouslySetInnerHTML`の使用禁止**: 生HTMLを挿入する機能は使用しない
- **サーバー側エスケープ**: Go側でも必要に応じてHTMLエスケープ

**安全な実装例**:
```typescript
// 安全（Reactが自動エスケープ）
<div>{article.title}</div>

// 危険（使用禁止）
<div dangerouslySetInnerHTML={{__html: article.summary}} />
```
